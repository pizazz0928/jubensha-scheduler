# 归来剧场 · 实时排班系统

面向盒装剧本杀门店的当日实时调度系统，重点处理临时开本、快速换 DM、DM 玩家补位、房间/人员冲突和操作留痕。

## 已实现

- 今日调度大屏、风险提醒、场次时间轴和 DM 实时状态
- 临时开本、主 DM 推荐、DM 玩家补位和快速换 DM
- 自定义 DM、剧本、熟练度、房间容量/类型/清理状态
- D1 数据读写、演示数据、操作日志和工作量统计
- 中文桌面界面和手机浏览器适配

## 本地开发

需要 Node.js 22.13 或更高版本。

```bash
pnpm install
pnpm dev
```

常用检查：

```bash
pnpm test
pnpm lint
pnpm build
```

## 腾讯云托管（CloudBase Run）

仓库根目录已包含 `Dockerfile`，腾讯云从 GitHub 拉取后可以直接构建。

建议配置：

- 构建目录：仓库根目录（不要填写 `dist`）
- Dockerfile：`Dockerfile`
- 服务端口：`3000`；运行时也会自动读取平台提供的 `PORT`
- 健康检查路径：`/healthz`
- 启动命令：留空，使用镜像内置命令

容器使用非 root 用户运行，并会把平台的停止信号传给应用，避免更新版本时进程无法正常退出。

### 重要的数据说明

当前腾讯兼容模式通过 Wrangler 本地运行时承载 Cloudflare D1 接口。它能让服务启动并完成数据读写，但数据写在容器本地目录；CloudBase Run 是无状态运行环境，容器重建、缩容或多实例运行后，本地数据可能丢失或不一致。

因此：

1. 演示阶段请把最大实例数设为 `1`，降低多实例数据分叉的风险。
2. 不要把容器本地数据当作正式备份。
3. 门店正式使用前，应迁移到外部持久数据库（例如腾讯云数据库/MySQL），或改为直接部署到带真实 D1 绑定的 Cloudflare Workers。

这不是 Docker 配置可以彻底解决的问题，而是托管平台的数据模型限制。

### 公开访问安全

当前版本尚未接入正式账号登录和角色权限。公开网址适合演示，不建议直接保存真实客人手机号、员工手机号或其他敏感资料。正式营业前应增加登录鉴权，并限制管理接口访问。

## 数据库

Cloudflare 环境下使用名为 `DB` 的 D1 绑定。表结构位于 `db/schema.ts`，迁移文件位于 `drizzle/`。

```bash
pnpm db:generate
```

基础资料接口位于 `/api/catalog`，调度日志接口位于 `/api/dispatch`。首次读取基础资料时会创建表并写入演示数据。

## 演示流程

1. 点击右下角“临时开本”，选择剧本并采用推荐方案。
2. 在缺 DM 的场次点击“安排 DM”，比较候选人的评分和影响。
3. 在缺玩家的场次点击“添加补位”，确认后玩家人数与 DM 状态会联动。
4. 在“DM 状态”“剧本管理”“房间管理”中维护自定义资料。
5. 再次临时开本时，系统会使用最新资料进行推荐。

演示数据包含 12 名 DM、15 个剧本、6 个房间和 8 个今日场次，并覆盖缺 DM、缺玩家、DM 补位、房间冲突和多次交接场景。

## 当前边界

第一版聚焦门店当日调度主流程。账号登录、细粒度权限、正式外部数据库、通知和工资结算仍属于后续阶段。
